# 嵌入式 AI 辅助开发提示词规范

> 版本：v1.0 | 维护人：seaman.player | 更新日期：2025-11  
> 适用工具：Cursor IDE / GitHub Copilot / Claude API / ChatGPT / DeepSeek  
> 适用场景：嵌入式 Linux、RTOS、音视频 SDK、驱动开发、跨平台移植

---

## 一、总则

### 1.1 规范目标

- 统一团队 AI 辅助开发提示词写法，减少低质量生成、减少返工。
- 明确 AI 可信任边界：哪些代码可直接采用，哪些必须人工验证。
- 将 Vibe Coding（意图驱动编程）落地为可执行的工程流程。

### 1.2 核心原则

| 原则 | 说明 |
|------|------|
| **上下文优先** | 提示词必须提供芯片型号、OS、工具链、SDK 版本等关键约束 |
| **意图明确** | 描述"要什么"而非"怎么做"，让 AI 做选型，工程师做验证 |
| **最小完整** | 单次提示只解决一个明确子问题，避免混合需求 |
| **可验证输出** | 要求 AI 同步输出测试用例或验证方法 |
| **安全红线** | 涉及中断、内存映射、电源管理的代码，AI 生成后必须人工逐行 Review |

## 二、提示词基础模板

### 2.1 通用结构（五段式）

```
[角色]  你是一名熟悉 {平台} 的嵌入式 C/C++ 工程师。
[背景]  当前项目：{项目名}，芯片：{芯片型号}，OS：{Linux/RTOS}，工具链：{GCC 版本}，SDK：{SDK 名称及版本}。
[约束]  {不能使用的库/函数}；{内存限制}；{实时性要求}；{编码规范，如 MISRA-C/Google Style}。
[任务]  {具体任务描述，动词开头，单一目标}。
[输出]  {代码 + 注释说明关键逻辑 + 简单测试用例 or 验证方法}。
```

**示例：**

```
[角色]  你是一名熟悉 RK3588 + Debian12 的嵌入式 Linux C 工程师。
[背景]  项目：户外 LED 大屏控制终端；芯片：RK3588；OS：Debian12；工具链：GCC 11.3；SDK：rkmedia 1.7。
[约束]  不使用 Qt；代码风格遵循 Linux Kernel Coding Style；函数不超过 80 行。
[任务]  实现一个通过 Modbus RTU 协议读取烟雾传感器寄存器的函数，支持超时重试（最多3次）。
[输出]  C 代码 + 每个关键步骤注释 + 简单的单元测试桩函数。
```

## 三、分场景提示词规范

### 3.1 驱动开发

**适用：** UART/I2C/SPI/GPIO 驱动、设备树适配、字符设备驱动

**必填上下文字段：**
- 内核版本（`uname -r` 结果）
- 芯片厂商 BSP 版本
- 硬件接口规格（引脚、频率、电气特性）
- 是否需要 DMA / 中断模式

**模板：**

```
芯片：{型号}，内核：{版本}，BSP：{版本}。
外设：{外设名称}，接口：{I2C/SPI/UART}，地址/速率：{值}。
要求：实现 {功能}，使用 {中断/轮询/DMA} 模式，兼容设备树配置。
输出：驱动 .c/.h 文件框架 + 对应 dts 节点示例 + probe/remove 函数注释。
```

**⚠️ 安全要求：** 以下内容 AI 生成后必须人工 Review：
- `ioremap` / `request_irq` / `dma_alloc_coherent`
- 中断处理函数中的共享资源访问
- 电源管理回调（suspend/resume）

---

### 3.2 音视频 SDK 集成

**适用：** ffmpeg 封装、rkmedia/MPP 调用、WebRTC 适配、GStreamer pipeline

**必填上下文字段：**
- ffmpeg/MPP/WebRTC 版本
- 编解码格式及分辨率帧率要求
- 是否使用硬件加速（VPU/GPU）
- 目标延迟指标

**模板：**

```
平台：{RK3588/Hisi/T40}，库：{ffmpeg 6.x / rkmedia / MPP}。
需求：实现 {H264/H265} {编码/解码}，分辨率 {1080P/4K}，帧率 {30/60}fps，{硬件加速/软件}。
延迟目标：{端到端 < XXms}。
输出：核心 API 调用代码 + 参数配置说明 + 常见错误处理。
```

---

### 3.3 WebRTC / RTC SDK 裁剪与移植

**适用：** WebRTC 嵌入式移植、信令对接、弱网优化

**必填上下文字段：**
- 目标芯片算力（MIPS/DMIPS 或实测空载 CPU 占用）
- 音视频编码格式需求
- 网络环境（弱网/内网/公网 P2P）
- 是否需要 NACK/FEC/BWE

**模板：**

```
目标平台：{芯片}，算力约束：{CPU 核心数，主频}，OS：{Linux/RTOS}。
WebRTC 版本：{M{版本号}}，需裁剪掉：{不需要的模块，如 VP8/VP9/DataChannel}。
音频：{采样率，通道数，AEC/ANS/AGC 开关}；视频：{编码格式，分辨率}。
输出：裁剪 GN 配置 + 关键 API 初始化代码 + 弱网参数推荐配置。
```

---

### 3.4 跨平台移植

**适用：** 源码从一个平台迁移到另一个平台（如从 Hisi 迁移到 RK）

**模板：**

```
源平台：{Hisi3518ev200}，目标平台：{RK3588}，OS 均为 {Linux}。
迁移内容：{模块名，如 H264 编码封装层}。
两平台差异点：{已知的 API 差异，如 MPI vs MPP}。
要求：保持对外接口不变（接口头文件已提供如下），仅替换底层实现。
输出：适配层代码 + 差异点注释 + 需人工确认的风险点清单。
```

---

### 3.5 CMake / Makefile 构建脚本

**模板：**

```
项目结构：{简述目录树}，目标产物：{so / elf / a}。
工具链：{交叉编译工具链前缀，如 aarch64-linux-gnu-}，sysroot：{路径}。
依赖库：{列出库名及版本}。
要求：支持 Debug/Release 切换，输出 compile_commands.json 供 Clangd 使用。
输出：CMakeLists.txt 完整内容 + 关键变量注释。
```

---

### 3.6 单元测试 / 桩函数生成

**模板：**

```
被测函数：{函数签名 + 简要说明}（代码如下）。
测试框架：{Unity / Google Test / 自定义桩}。
需覆盖场景：正常流程 / 边界值（{列出边界条件}）/ 异常返回。
输出：测试用例代码 + 覆盖率预估说明。
```

## 四、提示词质量检查清单

在向 AI 发送提示词前，自查以下项目：

- [ ] 是否说明了芯片型号和 OS 版本？
- [ ] 是否指定了关键库/SDK 版本？
- [ ] 是否说明了不能用的函数或库？
- [ ] 任务是否只有一个明确目标（没有 AND 并列多个任务）？
- [ ] 是否要求了输出格式（代码 + 注释 + 验证方法）？
- [ ] 是否标注了实时性 / 内存 / 功耗约束？

## 五、AI 生成代码 Code Review 标准

### 5.1 必须人工逐行验证的高风险点

| 类别 | 典型场景 | 风险 |
|------|---------|------|
| 内存操作 | `malloc`/`free`、DMA buffer、环形缓冲 | 内存泄漏、UAF |
| 中断上下文 | ISR 中的共享变量访问 | 竞态条件 |
| 电源管理 | suspend/resume 回调 | 死锁、数据丢失 |
| 硬件时序 | I2C/SPI 延迟、上电时序 | 硬件损坏 |
| 加密/认证 | TLS 配置、设备证书 | 安全漏洞 |
| 并发控制 | mutex/semaphore/spinlock 使用 | 死锁、优先级反转 |

### 5.2 审查流程

```
AI 生成代码
    │
    ├─ 1. 静态分析（cppcheck / clang-tidy）—— 自动
    ├─ 2. 高风险点人工逐行 Review（见 5.1 表）—— 必须
    ├─ 3. 单元测试运行（Unity / GTest）—— 自动
    ├─ 4. 目标板集成测试（功能验证）—— 必须
    └─ 5. 长稳测试（≥24h）—— 视模块重要性决定
```

### 5.3 Review 通过标准

- 无 cppcheck error 级别告警
- 高风险点人工确认无误
- 核心路径单测覆盖率 ≥ 70%
- 目标板跑通基本功能无崩溃

## 六、禁止直接使用 AI 生成代码的场景

以下场景 AI **只能辅助设计思路，代码必须由工程师手写或深度改写**：

1. **Bootloader / U-Boot 修改** — 风险：变砖
2. **量产烧录脚本** — 风险：批量损坏设备
3. **OTA 升级核心逻辑** — 风险：大规模变砖
4. **加密密钥管理 / eFuse 操作** — 风险：不可逆安全漏洞
5. **实时控制闭环（PID / 电机控制）** — 风险：硬件损坏 / 人身安全

## 七、工具配置推荐

### 7.1 Cursor IDE

```jsonc
// .cursor/rules（项目根目录）
{
  "rules": [
    "始终使用 C11 标准，禁止 VLA",
    "嵌入式代码不使用 C++ 异常",
    "函数超过 60 行需拆分",
    "所有 malloc 必须有对应 free 且在同一作用域明确",
    "中断函数命名以 _isr 结尾",
    "代码注释使用中文，变量/函数名使用英文"
  ]
}
```

### 7.2 GitHub Copilot 项目上下文文件

在项目根目录放置 `.github/copilot-instructions.md`：

```markdown
## 项目上下文
- 平台：RK3588 / Debian12
- 工具链：aarch64-linux-gnu-gcc 11.3
- 编码规范：Linux Kernel Coding Style
- 禁用：C++ 异常、RTTI、动态库（目标 .a 静态链接）
- 内存：堆内存分配必须有失败处理分支
```
